Deploy a Static Site to AWS (S3 + CloudFront + GitHub Actions)

Deployment 2026-07-06 中文版
TL;DR

Host the site in a private S3 bucket, serve it through CloudFront (CDN + HTTPS) via OAC. GitHub Actions authenticates with OIDC (no long-lived access keys); every merge to main runs aws s3 sync and invalidates the CloudFront cache. One-time setup, then it's just push-to-deploy.

Architecture

GitHub repo ──(merge to main)──▶ GitHub Actions
                                     │  OIDC assume role (no static keys)
                                     ▼
                              aws s3 sync site/ ──▶ S3 bucket (private)
                                     │                    ▲
                                     ▼                    │ OAC (CloudFront-only read)
                       cloudfront create-invalidation     │
                                                    CloudFront (HTTPS + CDN)
                                                          ▲
                                                        Users

Step 1: Create the S3 bucket

  1. S3 → Create bucket with a globally unique name, e.g. my-knowledge-base-site.
  2. Keep "Block all public access" enabled — CloudFront reads via OAC, the bucket never needs to be public.
  3. Do not enable static website hosting (that's only for the public website endpoint; OAC uses the regular REST endpoint).

Step 2: Create the CloudFront distribution (with OAC)

  1. CloudFront → Create distribution, origin = your S3 bucket.
  2. Origin access: Origin access control settings (OAC) → create a new OAC.
  3. Viewer protocol policy: Redirect HTTP to HTTPS.
  4. Default root object: index.html (otherwise the root URL returns 403).
  5. After creation, CloudFront shows a bucket policy to copy into S3 (Permissions → Bucket policy):
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "Service": "cloudfront.amazonaws.com" },
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::<BUCKET>/*",
    "Condition": {
      "StringEquals": {
        "AWS:SourceArn": "arn:aws:cloudfront::<ACCOUNT_ID>:distribution/<DIST_ID>"
      }
    }
  }]
}
For a custom domain: add an alternate domain name on the distribution, request an ACM certificate in us-east-1, then point DNS at the distribution with a CNAME/alias record.

Step 3: IAM — GitHub OIDC provider + deploy role

OIDC lets GitHub Actions assume a role directly — no AWS access keys stored in GitHub, nothing to leak or rotate.

3a. Create the OIDC identity provider (once per AWS account)

IAM → Identity providers → Add provider:

  • Provider type: OpenID Connect
  • Provider URL: https://token.actions.githubusercontent.com
  • Audience: sts.amazonaws.com

3b. Create the deploy role

IAM → Roles → Create role → Web identity, pick the provider above. The trust policy should be scoped to your repo and the main branch:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Federated": "arn:aws:iam::<ACCOUNT_ID>:oidc-provider/token.actions.githubusercontent.com"
    },
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": {
      "StringEquals": {
        "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
      },
      "StringLike": {
        "token.actions.githubusercontent.com:sub": "repo:<GITHUB_USER>/<REPO>:ref:refs/heads/main"
      }
    }
  }]
}

Permissions policy — least privilege, just sync + invalidation:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:ListBucket"],
      "Resource": "arn:aws:s3:::<BUCKET>"
    },
    {
      "Effect": "Allow",
      "Action": ["s3:PutObject", "s3:GetObject", "s3:DeleteObject"],
      "Resource": "arn:aws:s3:::<BUCKET>/*"
    },
    {
      "Effect": "Allow",
      "Action": ["cloudfront:CreateInvalidation"],
      "Resource": "arn:aws:cloudfront::<ACCOUNT_ID>:distribution/<DIST_ID>"
    }
  ]
}

Step 4: GitHub repo secrets

Repo → Settings → Secrets and variables → Actions, add four secrets:

SecretValue
AWS_DEPLOY_ROLE_ARNRole ARN from step 3b, e.g. arn:aws:iam::123456789012:role/kb-deploy
AWS_REGIONBucket region, e.g. ap-southeast-1
S3_BUCKETBucket name
CLOUDFRONT_DISTRIBUTION_IDe.g. E1A2B3C4D5E6F7

Step 5: GitHub Actions workflow

Lives at .github/workflows/deploy.yml (already in this repo):

name: Deploy to AWS

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  id-token: write   # required for OIDC
  contents: read

concurrency:
  group: deploy
  cancel-in-progress: true

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_DEPLOY_ROLE_ARN }}
          aws-region: ${{ secrets.AWS_REGION }}

      - name: Sync site/ to S3
        run: aws s3 sync site/ "s3://${{ secrets.S3_BUCKET }}" --delete

      - name: Invalidate CloudFront cache
        run: >
          aws cloudfront create-invalidation
          --distribution-id "${{ secrets.CLOUDFRONT_DISTRIBUTION_ID }}"
          --paths "/*"
  • permissions: id-token: write — OIDC fails without it.
  • --delete — files removed from the repo are removed from S3 too, keeping them in sync.
  • concurrency — rapid consecutive merges cancel the older deploy instead of racing.
  • Invalidations: first 1,000 paths per month are free; /* is the simplest choice for a personal site.

Verify

  1. Merge a PR into main (or hit Run workflow in the Actions tab).
  2. Both steps green in the Actions log.
  3. The CloudFront domain (https://dxxxx.cloudfront.net) serves the site; changes appear after the next merge.

Troubleshooting

SymptomCause / fix
403 AccessDenied at the root URLDefault root object not set to index.html, or bucket policy not applied
Not authorized to perform sts:AssumeRoleWithWebIdentityTrust policy sub doesn't match (repo/branch typo), or workflow missing id-token: write
Deploy succeeds but site unchangedCloudFront cache not invalidated — check the invalidation step ran
/foo/ doesn't serve its index.htmlDefault root object only applies at the root; add a CloudFront Function URI rewrite if needed (this site links full .html paths, so it's unaffected)
